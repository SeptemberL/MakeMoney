## Context

- 项目已具备股票数据抓取与入库能力，但主要用于常规更新，缺少“按缺失区间回补”的统一入口。
- 设置页面已承载运行参数管理，适合作为低频运维动作的触发入口。
- 用户需求是检查“每只股票当前库中最新日期”是否落后于最新交易日，若中间缺多天则补齐。

## Goals / Non-Goals

**Goals:**

- 提供一个设置页按钮，手动触发全量股票缺失日线回补任务。
- 以“每只股票最新日期 -> 最新交易日”的区间计算缺口，逐只补齐缺失天数据。
- 返回可诊断结果：处理股票总数、补齐股票数、补齐记录数、失败明细。

**Non-goals:**

- 不实现定时自动调度。
- 不引入新的数据库表结构。
- 不改造为复杂异步任务系统（V1 允许同步执行或轻量后台线程，取决于现有架构）。

## Decisions

1. **触发入口**
   - 在设置页面新增“补齐缺失日线数据”按钮。
   - 点击后调用后端 `POST` 接口触发任务，前端轮询或等待响应展示结果。
   - **理由**：满足“手动可控”，避免影响现有定时/日常流程。

2. **缺失判定规则**
   - 对每只股票读取数据库中该股票已有日线数据的 `max(trade_date)`。
   - 计算从 `max(trade_date)+1` 到“当前最新交易日”的区间。
   - 若区间非空则执行补齐；空区间则跳过。
   - **理由**：精确覆盖“中间差很多天”场景，避免全量重拉。

3. **交易日基准**
   - “当前最新交易日”优先使用现有交易日历/行情模块能力；若暂无交易日历能力，V1 以当日日期并结合数据源返回做兜底。
   - **理由**：减少非交易日误判。

4. **执行策略**
   - 逐股票处理，单只股票失败不影响整体继续。
   - 单次执行输出汇总与失败列表，便于重试。
   - **理由**：提升稳健性，满足运维可观测性。

5. **数据库一致性**
   - 本变更不涉及表结构修改，不触发表结构层面的 MySQL/SQLite 双引擎迁移要求。
   - 读写逻辑需保持对当前数据库类型兼容。

## Risks / Trade-offs

- **执行耗时较长**：股票数量大时一次补齐可能较慢；V1 可先串行，后续再优化并发与分批。
- **数据源限流**：连续请求可能触发限频；建议增加最小间隔或批次休眠策略。
- **重复写入风险**：若数据源返回重叠日期，需依赖已有 upsert/唯一键约束去重。

## Migration Plan

- 该功能为新增入口，无数据库迁移。
- 部署后在设置页可见新按钮，默认不自动执行。

## Open Questions

- 设置页当前请求模式是否支持长耗时接口（同步等待）？若不支持，应改为后台任务 + 状态查询接口。
- 最新交易日应统一以哪个模块输出为准（交易日历、行情源返回、或配置）？
