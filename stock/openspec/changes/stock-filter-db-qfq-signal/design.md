## Context

当前系统存在两类行情来源：

- 数据库落地的日线/行情数据：口径为**未复权**（raw）。
- 线上/实时拉取的数据：口径不固定（可能是前复权/后复权/未复权，取决于数据源与调用方式），并且可能绕过数据库存储层被直接用于信号计算。

这会导致同一套指标/信号在不同入口下计算结果不一致，影响筛选、告警与回溯分析。根据 proposal，本次变更的目标是：**信号计算统一使用数据库数据**，并在计算前把未复权行情按库内“前复权因子”转换为**前复权口径**。

约束与假设：

- 数据库中存在可按（股票、交易日）获取的**前复权因子**，来源为 `adj_factor` 表（按 `ts_code + trade_date` 查询/批量查询）。
- K 线数据与复权因子需要按交易日严格对齐；若缺失，需要显式策略。
- 需要兼顾性能（批量计算/筛选场景）与可观测性（口径可追踪）。

## Goals / Non-Goals

**Goals:**

- 将信号计算的数据源收敛为：DB 行情（raw） + DB 前复权因子 → 计算前复权序列 → 指标/信号计算。
- 在计算结果或日志中明确标注本次计算的**数据来源与复权口径**，便于排错与回放。
- 对“缺失复权因子 / 缺失交易日行情 / 对齐失败”给出确定性处理策略，避免静默错误。
- 在批量筛选场景中保持可接受的性能：复权因子与行情尽量批量读取/缓存。

**Non-Goals:**

- 不在本次变更中引入新的行情供应商或改造线上实时行情接口协议。
- 不保证所有历史信号结果与新口径完全可比（口径变化是预期影响）；仅保证新计算链路一致且可追溯。
- 不在本次变更中实现全量历史信号重算（如需重算可作为后续任务/运维脚本）。

## Decisions

- **统一信号计算入口的数据源为数据库**
  - **Decision**：`stock_filter` 与信号检查系统（包括 `price-level-interval-signal`）在计算信号时，默认只从数据库读取行情与复权因子。
  - **Rationale**：避免线上数据口径不透明导致的非确定性；数据库是可回放、可审计的事实来源。
  - **Alternatives**：
    - 允许线上数据参与并做口径推断：实现复杂且易出错；一旦数据源策略变化会漂移。
    - 将数据库直接改存前复权行情：会引入数据迁移与存储膨胀，并丢失 raw 事实口径，不利于以后切换复权方式。

- **复权转换策略：使用“前复权因子”对价格字段做乘法缩放**
  - **Decision**：读取 raw 行情后，通过（同交易日的）前复权因子将 `open/high/low/close` 转为前复权口径；`volume/amount` 是否转换由指标需求决定（默认不变或按既定口径处理），并在 spec 中约束。
  - **Rationale**：前复权适合技术分析指标计算与价格形态对比；保持 raw 作为底层事实数据，转换在计算侧完成。
  - **Alternatives**：
    - 使用后复权：对历史回放/实时拼接会有不同的语义；本次明确前复权。
    - 直接查询第三方返回的前复权 K 线：引入外部依赖与不稳定性，且不利于离线批量。

- **对齐与缺失处理：明确失败即显式降级/跳过**
  - **Decision**：复权因子缺失或交易日无法对齐时，不允许“默默按未复权继续算且当作前复权”。采用可配置策略：
    - 默认：标记该股票/该区间“不可计算”并跳过该信号产出（或产出 `status=insufficient_data`）。
    - 可选：回退到未复权计算，但必须在输出中标注 `adjustment=raw_fallback`。
  - **Rationale**：避免 silent data corruption；在筛选/告警中宁可缺失也不要错误命中。
  - **Alternatives**：插值/近似因子填补（风险高，且会放大误差）。

- **性能：批量读 + 缓存复权因子**
  - **Decision**：对批量筛选/全市场扫描：按（交易日区间、股票集合）批量读取行情与复权因子；在单次任务执行周期内对复权因子做内存缓存（按 `ts_code + trade_date` 或按区间）。
  - **Rationale**：复权因子表通常体量小于行情，且访问模式集中；缓存可显著减少数据库往返。
  - **Alternatives**：逐股票逐日查询（实现简单但性能不可接受）。

- **可观测性：在信号输出与日志中统一埋点口径字段**
  - **Decision**：信号计算结果结构（或任务日志）增加字段：`data_source=db`、`adjustment=qfq|raw|raw_fallback`、`factor_coverage=complete|partial|missing` 等。
  - **Rationale**：支持问题定位与线上对账；未来引入实时信号也能明确区分口径。

## Risks / Trade-offs

- **[复权因子表不存在/不完整]** → 先补齐数据生产链路或提供“跳过/回退”策略，并在任务输出中强制暴露覆盖率。
- **[交易日对齐复杂（停牌/缺失日）]** → 以交易日为主键对齐；区间计算时只使用两表交集，缺口触发策略而非隐式补齐。
- **[性能回退]** → 批量查询 + 缓存；必要时为复权因子表建立索引（`ts_code, trade_date`）。
- **[历史结果不可比]** → 在文档与输出中标注口径升级；需要时提供一次性重算工具（非本次必做）。
