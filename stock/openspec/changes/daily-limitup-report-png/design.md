## Context

当前涨停复盘任务（`daily_limitup_report`）已能产出结构化 JSON（例如 `outputs/daily_limitup_report/<date>/report.json`），适合程序消费但不适合快速浏览与转发分享。需要将同一份数据渲染为统一风格的 PNG 图片（表格/卡片形式），用于日常复盘、群聊分享与留档。

约束：
- 图片需要支持中文字体渲染（股票名称、字段标题）。
- 输出应可离线生成（不依赖浏览器交互），并可在定时任务中运行。
- 输出规模可能较大（多只股票），需要考虑分页/分片生成与文件命名。

## Goals / Non-Goals

**Goals:**
- 从指定交易日的涨停复盘 JSON 输入，生成一个或多个 PNG 图片文件。
- PNG 图片至少包含：交易日、生成时间、股票列表以及关键字段（连板、首次涨停、最终封板、首次换手、收盘换手）。
- 支持“分页输出”（例如每张 30 行），并保证输出路径可预测、可覆盖重跑。
- 生成逻辑提供命令/任务入口，便于定时执行或手动补跑。

**Non-Goals:**
- 不在本阶段实现复杂交互式图表（K 线、成交量分布等）。
- 不强绑定具体推送渠道（微信/飞书）的一键发送；如需可在后续扩展为可选后置步骤。

## Decisions

1) **渲染技术选型**
- 选择：优先使用 Pillow（PIL）进行纯 Python 渲染（绘制标题、表格、文本与简单色块）。
- 理由：依赖更轻、适合离线任务、无需浏览器；易于控制输出尺寸与字体。
- 备选：matplotlib（表格渲染）或 HTML+Playwright（更漂亮但依赖重、运行环境复杂）。

2) **分页与版式**
- 选择：默认每张 PNG 固定行数（例如 30），超出则分页输出 `page_1.png`、`page_2.png`。
- 理由：避免超长图片影响阅读与发送；输出更可控。

3) **输入数据契约**
- 选择：以 `daily_limitup_report` 的 `report.json` 为唯一输入，字段缺失时按 `status/reason` 以占位显示（例如 `--`）。
- 理由：复用现有任务产物；对缺失分钟数据可自然降级显示。

## Risks / Trade-offs

- **[中文字体缺失导致乱码/方块]** → 内置字体探测与回退策略（例如系统字体路径优先，缺失则提示并降级为 ASCII）。
- **[行数过多导致图片过长]** → 默认分页 + 可配置每页行数与字体大小。
- **[字段较多导致拥挤]** → 固定列宽 + 省略策略（例如名称截断）+ 可配置显示列集合。
