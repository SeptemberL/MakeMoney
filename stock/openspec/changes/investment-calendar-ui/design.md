## Context

当前项目是基于 Flask + 模板页面的 Web 控制台（见 `main.py` 注册 `routes` 蓝图）。已有多处“管理/控制台”类页面能力（例如定时任务、设置、信号通知等），但缺少一个围绕“日期”组织投资事项的入口。

本变更引入“投资日历”月视图（万年历），并把日历项按账号持久化存入账号数据库（account DB）。由于项目同时支持 MySQL 与 SQLite，涉及表结构变更时需要保证双引擎语义一致。

## Goals / Non-Goals

**Goals:**
- 提供按月展示的万年历页面，并支持跨月/跨年浏览。
- 点击某日打开弹窗：创建/编辑/删除日历项；弹窗内展示该日期下的已添加列表，点击可加载编辑。
- 日历项按账号隔离存储到账号数据库（MySQL/SQLite 均可），并提供稳定的 CRUD 服务接口供 UI 调用。
- 数据校验清晰可预期：日期格式、字段长度、空值处理、并发编辑的基本保护。

**Non-Goals:**
- 不在本次设计中实现“到点自动推送提醒”（仅保存提醒信息；真正的提醒触发可后续接入现有 scheduler/notify 通道）。
- 不引入第三方日历同步（如 iCal、Google Calendar）。
- 不做复杂重复规则（RRULE）与多时区/小时级日程；以“按日”记录为核心。

## Decisions

### 1) UI 实现：月历视图 + 日期弹窗（单页内完成 CRUD）

- **选择**：新增一个独立页面（例如 `/investment-calendar`）渲染月历；点击日期时打开模态弹窗，在弹窗内完成：
  - 表单：内容（content）、提醒消息 group（reminder_group）、提醒消息内容（reminder_message）
  - 列表：该日期的日历项列表（可点击加载到表单）
- **理由**：日历是强交互组件，弹窗内完成 CRUD 可以减少页面跳转、提升“当天快速维护”效率。
- **备选**：拆成“日列表页 + 编辑页”。缺点是跳转多、编辑成本高。

### 2) 后端接口：面向日期范围的查询 + 单条 CRUD

- **选择**：提供最小完备的一组 HTTP API（具体路径/方法在 specs 中定稿）：
  - 按月查询：给定 `month`（或 `start/end`），返回日期 -> items 列表（便于月历标记与弹窗列表）
  - 单条创建/更新/删除：返回最新实体，便于 UI 即时刷新
- **理由**：月视图需要“整月聚合”；弹窗编辑需要“单条写入”。
- **备选**：每次点击日期再拉取当天列表。缺点是交互上更慢、请求次数更多。

### 3) 数据模型：账号库新增“按日的日历项”表（单表承载提醒字段）

- **选择**：使用一张表存储日历项，字段包含：
  - `account_id`（账号隔离）
  - `date`（YYYY-MM-DD，按日）
  - `content`
  - `reminder_group`
  - `reminder_message`
  - `created_at` / `updated_at`
- **理由**：当前需求中每个日历项仅需要一个 group + 一段提醒内容；单表实现简单、查询高效。
- **备选**：规范化为“item 表 + reminders 子表”。当未来需要“一个日历项多个提醒”或“提醒触发时间”时更灵活；本次先不引入复杂度。

### 4) 账号隔离与权限：所有读写都必须绑定当前账号上下文

- **选择**：所有 CRUD 以当前账号上下文（例如 session/配置/登录态）确定 `account_id`，任何 API 参数不得越权指定其他账号。
- **理由**：日历属于个人资产；账号隔离是最重要的安全边界。
- **备选**：全局共享表。会导致多账号数据混杂，不满足“保存在账号数据库”的要求。

## Risks / Trade-offs

- **[Risk] 双引擎表结构不一致** → **Mitigation**：所有表/索引/约束变更必须同时更新 MySQL 与 SQLite 的创建/迁移逻辑；在 tasks 中明确列出双引擎检查点。
- **[Risk] 月视图查询性能**（整月返回数据过多）→ **Mitigation**：以 `date` + `account_id` 作为索引关键；限制字段长度；按月范围查询并分页预留（先不实现分页，但 API 结构可扩展）。
- **[Risk] 并发编辑覆盖**（两处同时改同一条）→ **Mitigation**：返回 `updated_at`，更新时可带 `if_match_updated_at`（或乐观锁版本号）作为可选参数；不做强一致但可检测冲突。
- **[Trade-off] 单表提醒字段的扩展性不足** → **Mitigation**：spec 中把提醒字段定义为“可选”；后续如需多提醒/定时触发，可新增子表并通过迁移兼容旧数据。
