## Context

项目当前已有“通知/群”概念（例如信号规则中存在 `group_ids`），但缺少飞书作为一个可配置的通知渠道；同时现有的“群”配置无法表达“同一个 group 走 OAPI 或 HTTPS 两种不同发送实现”的差异，也缺少可视化的设置入口来维护多组配置。

本变更引入官方 `lark-oapi` 作为 OAPI 实现，并在配置层保留 HTTPS 发送方式作为兼容/兜底（例如在某些网络环境下 OAPI 不可用或需要快速临时切换实现）。

约束：
- 配置体验需对齐现有“微信”设置模式：以 `group` 为中心的多项列表配置
- 每个 group 必须能独立选择发送方式（`oapi` / `https`）
- 通知渠道的选择由设置中的**全局开关**决定：全局选飞书则解析飞书 groups；全局选微信则解析微信配置
- 不要求改变信号计算逻辑；信号/业务侧继续只传 `group_id`（或 `group_ids`），发送端根据全局开关决定走微信还是飞书

## Goals / Non-Goals

**Goals:**
- 在设置界面提供“飞书设置”，支持维护多个飞书 group
- 每个飞书 group 支持选择发送方式：`oapi`（基于 `lark-oapi`）或 `https`（Webhook/HTTP）
- 通知系统在发送时按如下两层路由：
  - 先根据全局通知渠道（微信/飞书）选择“从哪里加载 group 配置”
  - 若渠道为飞书，再根据 group 的 `send_mode` 选择 OAPI/HTTPS sender
- 统一两种发送方式的基础能力：超时、错误信息、最小可用日志，便于定位问题

**Non-Goals:**
- 不在本变更中重构所有通知渠道的架构（仅新增飞书渠道并做最小必要的抽象）
- 不在本变更中引入复杂的消息编排（富文本、卡片模板编辑器等）作为硬性目标
- 不改变既有信号触发条件、节流逻辑与持久化结构（除非实现过程中发现必要最小改动）

## Decisions

1) 以 “group” 作为飞书发送配置的第一类实体
- **选择**：在配置中新增 `feishu.groups[]`（或等价结构），每项代表一个可发送目标
- **原因**：贴合用户心智（像微信一样配置 group），也便于信号规则沿用 `group_ids` 的选择模式
- **备选方案**：仅配置一个全局 webhook / 全局 app，无法满足“多 group + 每个 group 独立发送方式”的需求

2) 业务侧保持 `group_id` 不变，由全局通知渠道决定解析来源
- **选择**：信号规则与其他业务功能继续只保存 `group_id`（或 `group_ids`）；运行时根据设置的全局通知渠道决定从“微信 groups”还是“飞书 groups”加载对应配置
- **原因**：最小化对现有业务/规则/接口的改动；切换渠道时无需改动大量规则配置，只需切换全局开关并提供同名/同 id 的 group 配置
- **备选方案**：在规则中显式区分 `wechat_group_ids` / `feishu_group_ids`（侵入性更强、迁移成本更高，不符合本次目标）

3) （飞书渠道内）发送方式按 group 维度选择（`send_mode`)
- **选择**：每个 group 保存 `send_mode: oapi|https`，并携带该模式所需的最小字段
- **原因**：实现上可直接通过策略模式/工厂路由到不同 sender；运维上可以快速切换某个 group 的实现而不影响其他 group
- **备选方案**：全局开关（过于粗粒度）、或在代码中写死（不可配置）

4) 两个 sender：OAPI sender 与 HTTPS sender
- **OAPI sender**：依赖 `lark-oapi`，负责 token 获取与调用飞书消息发送接口（以 chat_id 等标识发送）
- **HTTPS sender**：基于 `requests`（或项目既有 HTTP 客户端）直接 POST 到 webhook URL（或自定义 endpoint）
- **原因**：保持实现隔离、便于测试与切换；也能在未来扩展更多 sender（例如企业微信、钉钉）

5) 设置界面与配置落盘
- **选择**：沿用项目现有配置体系（`config.ini` + `configs/*.yaml` 等）进行扩展；设置界面做结构化表单，最终落盘为可读配置
- **原因**：符合当前项目形态；便于在无 UI 的环境中直接编辑配置并部署
- **备选方案**：引入数据库存储设置（需要迁移与双引擎同步，成本更高，不是本变更的必需）

## Risks / Trade-offs

- **[凭证泄露风险]** → 仅在本地/部署环境的配置文件中保存 app 凭证；避免将敏感信息写入日志；文档中明确不要提交配置到 git
- **[OAPI token/接口异常导致发送失败]** → sender 需要明确的超时与错误包装；允许切换到 HTTPS 作为兜底；记录可定位的错误码/响应摘要
- **[两种方式的消息格式差异]** → 在能力层定义“最小公共消息格式”（文本为主）；富格式能力留作后续扩展
- **[配置复杂度上升]** → 设置界面提供校验（必填字段、模式字段互斥）；加载配置时也做一次校验并给出清晰错误
