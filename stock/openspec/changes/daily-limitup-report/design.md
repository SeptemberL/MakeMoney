## Context

本变更目标是在每日收盘后（17:00 以后）对“当日涨停股票”做一次离线复盘计算，输出关键事件时刻与换手率指标，供后续展示、导出或策略筛选使用。

当前系统已存在若干与行情/自选相关的能力（如 `watchlist-intraday-metrics`），但缺少面向“涨停事件链路（触板/开板/回封/封死）”的统一口径与每日汇总输出。

约束与前置：
- 数据源需要提供：交易日历、日线（含是否涨停/收盘价）、分钟级或更细粒度的价格/成交（用于识别涨停时刻与开板回封）、换手率或可推导换手率所需字段（成交量、自由流通股本等）。
- 任务运行时间为 17:00 以后，允许 T+0 日内数据在收盘后补齐与修正。
- 若需要落库持久化，必须保持 MySQL 与 SQLite 表结构与迁移逻辑一致。

## Goals / Non-Goals

**Goals:**
- 在每日 17:00 后生成“当日涨停复盘结果集”，覆盖当日所有涨停个股。
- 对每只股票输出：
  - 连续涨停天数（连板数）
  - 当日首次涨停时刻
  - 当日最终封死涨停时刻（当天最后一次回到涨停后，直到收盘不再打开）
  - 首次涨停时刻的换手率
  - 收盘时的换手率
- 给出可测试、可复现的口径（尤其是“最终封死涨停”的判定）。
- 输出格式结构化（JSON/CSV/表），并可选持久化保存，支持幂等重跑。

**Non-Goals:**
- 不在本阶段实现盘中实时推送/预警（仅收盘后离线计算）。
- 不处理 ST、北交所、不同涨跌幅制度的全量适配细节（如需支持，作为后续扩展）。
- 不在设计里绑定具体第三方数据提供商的细节（仅定义所需字段与抽象接口）。

## Decisions

1) **当日涨停股票集合的确定**
- 选择：以日线结果为准（当日收盘价达到涨停价/涨停标记），生成候选列表；再在分钟数据上计算“首次触板/最终封板”等事件指标。
- 理由：日线能稳定定义“当日涨停”集合，分钟数据用于事件链路，避免仅靠分钟数据导致漏判或边界复杂化。
- 备选：仅基于分钟数据判定“触及涨停且收盘仍涨停”。缺点是边界条件多、对数据缺失更敏感。

2) **涨停事件的判定与时间戳**
- 选择：以“价格达到涨停价（limit_up_price）”作为触板；以“价格从涨停价跌破（< limit_up_price）”作为开板；以“最后一次触板后直到收盘未再开板”定义“最终封死涨停”时间。
- 理由：该定义可在分钟级 K 线或逐笔数据上计算，且可测试。
- 备选：使用数据源提供的“封板/开板”事件。优点省计算，缺点是可移植性差。

3) **换手率口径**
- 选择：优先使用数据源直接提供的分钟级/日级换手率；若无分钟换手率，则以累计成交量与自由流通股本推导（累计成交量 / free_float_shares）。
- 理由：减少依赖复杂复权与单位换算；在缺失分钟换手率时仍可计算。
- 备选：以成交额/流通市值近似，误差更大。

4) **连板天数（连续涨停天数）口径**
- 选择：以交易日为单位向前回溯，统计截至当日的连续“收盘涨停”天数；遇到非涨停日即停止。
- 理由：与“连板”常用口径一致，便于筛选。
- 备选：按“当日曾涨停”回溯，会与市场常用连板口径偏离。

5) **任务形态与幂等**
- 选择：提供一个可被调度系统调用的命令/任务入口（例如 `daily_limitup_report --date=YYYY-MM-DD`），默认处理最近交易日；结果按（交易日, 股票代码）幂等 upsert。
- 理由：便于补跑与重算；失败可重试；支持回溯历史。

## Risks / Trade-offs

- **[分钟数据缺失或延迟]** → 允许任务在 17:00 后运行并可重跑；对缺失股票输出状态为 `partial` 并记录缺失原因。
- **[不同市场/制度涨停价计算差异]** → 将“涨停价计算/获取”封装为接口；首版仅覆盖 A 股主板常规规则，后续扩展。
- **[最终封板判定受数据粒度影响]** → 明确以分钟粒度为准（或逐笔）；并在 spec 中声明粒度要求与误差边界。
- **[落库引入双引擎迁移复杂度]** → 若落库，迁移脚本必须同时更新 MySQL 与 SQLite；优先选择最小表结构、可扩展字段（JSON）以降低频繁变更。
