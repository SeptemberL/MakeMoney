## Context

项目为 Flask + Jinja2 的股票分析系统，已有本地行情（MySQL/SQLite）、技术指标（TA-Lib/MyTT）与 `/quant` 量化页；其中「策略回测」卡片仍为占位。本变更在**不破坏现有页面与 API** 的前提下，引入可运行的回测能力与清晰入口，并配套用户文档。

## Goals / Non-Goals

**Goals:**

- 提供**最小可用**的回测闭环：选择标的与时间区间、触发回测、展示核心结果（收益、回撤、交易统计等，具体以 spec 为准）。
- 复用现有**数据访问层**拉取 K 线；策略侧优先用 Python 向量化或清晰循环，避免重复造行情管道。
- 在 **`/quant`** 上提供从「策略回测」卡片到回测 UI 的**显式入口**（按钮或链接）。
- 在 **`Docs/`** 下新增回测说明文档，说明功能边界、操作步骤与限制。

**Non-Goals:**

- 本期不承诺多用户隔离的云端回测集群、实时模拟盘或全市场分钟级超高性能回测。
- 不强制实现完整的「策略 DSL 编辑器」；可先支持固定策略集或有限参数，后续迭代扩展。
- 因子库、组合优化等量化页上其他卡片不在本期交付范围。

## Decisions

1. **回测引擎位置**：在仓库内新增独立模块（例如 `backtest/` 或 `Quant/` 下子包），由路由层调用；避免把大量数值逻辑堆在 `routes.py`。
2. **数据与对齐**：回测 MUST 使用与项目一致的日线（或现有最小粒度）数据源；缺失数据时行为在实现中明确（跳过、报错或截断），并在文档中说明。
3. **Web 形态**：新增专用页面路由（例如 `/quant/backtest` 或 `/backtest`），与 `quant.html` 入口一致；模板风格与现有 Bootstrap 导航保持一致。
4. **API 形态**：优先 **POST/GET JSON API** 触发回测并返回结构化结果，页面用 fetch 或表单提交渲染；便于后续自动化测试。
5. **持久化**：首版**可不落库**（内存计算 + 页面展示）；若产品需要保存历史任务，再引入表结构，且须 **MySQL 与 SQLite 同步**（见风险缓解）。
6. **文档**：新增 `Docs/量化回测系统.md`（或提案阶段最终文件名），并在 `Docs/项目基础框架.md` 中增加一行路由说明（可选，由 tasks 执行）。

**备选方案**：纯静态页 + 前端计算 — 放弃，因与现有后端行情与指标栈不一致且难以维护。

## Risks / Trade-offs

- **[Risk] 回测结果与实盘差异** → **Mitigation**：文档标明假设（佣金、滑点、停牌、复权）；默认参数保守或显式展示。
- **[Risk] 大数据量请求阻塞 Web 进程** → **Mitigation**：限制日期跨度与标的数量；必要时后续加异步任务（非本期必做）。
- **[Risk] 双引擎表结构不一致** → **Mitigation**：若加表，严格按仓库规范同时改 MySQL 与 SQLite 迁移/建表路径。

## Migration Plan

- **部署**：代码与模板上线即可；无库表变更时无需数据迁移。
- **回滚**：移除新路由与模板入口即可恢复占位行为；若有 DB 变更则保留向下兼容迁移或提供回滚脚本。

## Open Questions
