## 阶段 2b.2：持仓与交易录入设计说明（极简版）

> 目标：在现有持仓只读视图基础上，增加“添加操作”入口与历史/明细页面，完成阶段 2b.2 的验收，而不引入复杂资金/仓位管理逻辑。

---

### 一、现状与目标

- **已存在的数据表（MySQL / SQLite 双引擎已实现）**
  - `positions`：当前持仓快照。
  - `transactions`：买入/卖出交易流水。
  - `portfolio`：组合概览（目前可视为预留，阶段 2b.2 极简模式下不强依赖）。

- **已存在的后端逻辑**
  - `GET /positions` → `positions_page()`：渲染 `positions.html`，通过 `_get_positions_with_market()` 获取：
    - 持仓基础字段：`id, stock_code, stock_name, quantity, cost_price`
    - 来自 `stock_data` 的最新 `close` 作为当前价
    - 计算字段 `total_value = quantity * current_price`
  - `_apply_position_trade(db, stock_code, stock_name, action, price, quantity, fee, trade_date)`：
    - 在一个已开启事务的连接上：
      - 插入一条 `transactions` 记录；
      - 根据 `action` 对 `positions` 做加/减仓更新：
        - `buy`：按加权平均成本更新 `quantity` 和 `cost_price`；
        - `sell`：校验卖出量不大于持仓量，减仓或清仓。
  - `POST /api/positions/trade` → `api_position_trade()`：
    - JSON 入参：`{stock_code, stock_name?, action, price, quantity, fee?, trade_date?}`
    - 做参数校验后，开启事务调用 `_apply_position_trade()`，返回 JSON 结果。
  - `GET /api/positions/<stock_code>/history` → `api_position_history()`：
    - 查询 `transactions` 中指定股票的所有流水，按 `trade_date DESC, id DESC` 排序，并返回 JSON 列表。

- **阶段 2b.2 的补全目标**
  - 在 `/positions` 页面增加“添加操作”按钮，进入表单页面，支持录入单笔买入/卖出。
  - 实现“历史池子页面”：展示所有交易流水。
  - 实现“单票操作明细页面”：展示单只股票的交易明细。

---

### 二、业务简化假设（极简模式）

为降低实现复杂度、专注阶段 2b.2 的验收，暂采用如下业务简化：

- **资金账户**：不做约束，视为“资金无限”；仅记录买卖金额，不校验可用资金。
- **撮合规则**：不考虑 T+1 / 融资融券等规则。
- **成本价**：
  - 买入时按加权平均成本更新 `positions.cost_price`。
  - 卖出时不改变 `cost_price`，仅减少 `quantity` 或清仓。
- **手续费 `fee`**：
  - 可选字段，默认 0。
  - 参与交易金额计算：`amount = price * quantity + fee`。
- **仓位关闭**：
  - 当卖出后 `new_qty <= 0`，直接从 `positions` 中删除该股票，历史信息保留在 `transactions`。

这些简化保证阶段 2b.2 可快速落地，同时为之后的“阶段三：仓位管理系统”预留扩展空间。

### 三、“添加操作”页面设计

#### 3.1 URL 与路由

- `GET /positions/trade`
  - 展示“添加操作”表单页面：`positions_trade.html`。
- `POST /positions/trade`
  - 接收表单提交，调用 `_apply_position_trade()` 完成买入/卖出逻辑，成功后重定向到持仓或历史页面。

#### 3.2 表单字段（HTML 层）

表单使用传统 POST 提交：

- `method="post"`, `action="/positions/trade"`

字段清单：

- **必填**
  - `stock_code`：股票代码
    - 类型：`text`
    - 示例：`000001`
  - `action`：买入 / 卖出
    - 类型：`select` 或 `radio`
    - 取值：`buy` / `sell`
  - `price`：成交价
    - 类型：`number`
    - 建议属性：`step="0.01"`
  - `quantity`：成交数量
    - 类型：`number`
    - 可以视需求设置 `step="1"`（整手）或支持小数。

- **可选**
  - `stock_name`：股票名称
    - 类型：`text`
    - 若为空，可由后端保留原值或通过 `stocks` 表补全。
  - `fee`：手续费
    - 类型：`number`
    - 缺省值：`0`，`step="0.01"`
  - `trade_date`：成交日期
    - 类型：`date`
    - 若为空：默认使用当前日期 `datetime.now().strftime('%Y-%m-%d')`
  - `next`：重定向目标（可选）
    - 类型：`hidden`
    - 可能的值：`positions`、`transactions` 等，便于提交后跳不同页面。

#### 3.3 后端处理逻辑（POST）

路由：`/positions/trade`，方法 `POST`。

推荐处理流程：

1. **读取表单参数（`request.form`）**
   - `stock_code = request.form.get('stock_code', '').strip()`
   - `stock_name = request.form.get('stock_name', '').strip() or None`
   - `action = (request.form.get('action') or '').lower()`
   - `price = request.form.get('price')`
   - `quantity = request.form.get('quantity')`
   - `fee = request.form.get('fee', 0)`
   - `trade_date = request.form.get('trade_date')`
   - `next_target = request.form.get('next', 'positions')`

2. **参数校验**
   - `stock_code` 不能为空。
   - `action` 必须为 `buy` 或 `sell`。
   - `price` / `quantity` / `fee` 需可转换为 `float`。
   - `price > 0` 且 `quantity > 0`。
   - `trade_date` 为空则填入当天日期。
   - 对于卖出操作，数量与持仓关系的校验交由 `_apply_position_trade()` 中完成：
     - 若无持仓或数量超出持仓，将抛出 `ValueError`，由路由捕获并在页面展示错误。

3. **执行业务逻辑**
   - `db = Database.Create()`
   - `db.begin_transaction()`
   - 调用 `_apply_position_trade()`：
     - 先插入 `transactions` 一条流水记录；
     - 按当前 `action` 更新 `positions`：
       - 买入：加权平均成本更新；
       - 卖出：减仓或清仓。
   - 成功：`db.commit()`；异常：`db.rollback()`。

4. **返回结果**
   - 成功：
     - 根据 `next_target` 选择重定向：
       - 默认：`redirect(url_for('main.positions_page'))`（即 `/positions`）。
       - 或者：跳到历史页面 `/positions/transactions`。
   - 失败：
     - 把错误原因写到 `error_message` 模板变量中并重新渲染 `positions_trade.html`，或用 `flash` 信息 + 重定向回表单页面。

#### 3.4 模板变量（`positions_trade.html`）

模板渲染建议提供如下变量：

- `today`：默认日期，字符串格式 `YYYY-MM-DD`，供 `input[type=date]` 使用。
- `default_action`：默认操作类型，例如 `'buy'`。
- `error_message`（可选）：上一次提交的错误提示，展示在页面顶部。
- `default_stock_code`、`default_stock_name`（可选）：从其他页面跳转过来时，可提前填入。

### 四、“历史池子页面”设计（所有交易列表）

#### 4.1 URL 与路由

- `GET /positions/transactions`
  - 渲染 `positions_transactions.html`。
  - 展示所有 `transactions` 记录，按时间倒序。

#### 4.2 数据查询

后端使用 `Database.fetch_all()` 查询：

- SQL 示例（兼容 MySQL / SQLite）：

  - `SELECT stock_code, action, price, quantity, fee, trade_date, created_at FROM transactions ORDER BY trade_date DESC, id DESC`

- 在 Python 侧为每条记录增加衍生字段：

  - `amount = price * quantity + fee`
  - 可额外增加：
    - `direction_label = '买入' / '卖出'`

#### 4.3 模板变量（`positions_transactions.html`）

渲染时建议提供：

- `transactions`：列表，每个元素包含：
  - `stock_code`
  - `action`（`buy` / `sell`）
  - `price`（float）
  - `quantity`（float）
  - `fee`（float）
  - `trade_date`
  - `created_at`
  - `amount`（float，`price * quantity + fee`）

可选扩展：

- `filters`：记录当前页面的筛选条件（如股票代码、日期范围），方便之后扩展查询表单。

#### 4.4 页面交互建议

在表格中，为每条交易增加一个指向单票明细页面的链接：

- 链接 URL：`/positions/{{ tx.stock_code }}/history_page`
- 显示文案：可以是“查看该票明细”或直接让股票代码本身可点击。

页头可加入简单统计信息（可后续扩展）：

- 总记录数、最近交易日期等。

### 五、“单票操作明细页面”设计

#### 5.1 URL 与路由

- `GET /positions/<stock_code>/history_page`
  - 渲染 `position_history.html`。
  - 展示指定 `stock_code` 在 `transactions` 表中的全部交易记录。

#### 5.2 数据查询与汇总

1. **基本交易记录**

   - SQL 示例：
     - `SELECT stock_code, action, price, quantity, fee, trade_date, created_at FROM transactions WHERE stock_code = %s ORDER BY trade_date DESC, id DESC`
   - 同样可以在 Python 侧为每条记录计算：
     - `amount = price * quantity + fee`

2. **汇总信息（可选，但推荐）**

   - 通过对该股票所有流水进行简单聚合，计算：
     - `total_buy_quantity`：买入总数量
     - `total_sell_quantity`：卖出总数量
     - `net_quantity = total_buy_quantity - total_sell_quantity`：理论当前持仓数量
     - `total_buy_amount`：买入总金额（含手续费）
     - `total_sell_amount`：卖出总金额（含手续费）
   - 与 `positions` 中该股票的当前 `quantity` 做对照：
     - 若不一致，可辅助排查录入错误。

3. **股票名称获取（可选）**

   - 若希望在页面标题中展示股票名称，可在后端额外查询一次：
     - 优先从 `positions` 表查：
       - `SELECT stock_name, quantity, cost_price FROM positions WHERE stock_code = %s`
     - 若无结果，可从 `stocks` 表查：
       - `SELECT name FROM stocks WHERE code = %s`

#### 5.3 模板变量（`position_history.html`）

建议提供：

- `stock_code`：当前股票代码。
- `stock_name`：股票名称（若可获取）。
- `records`：该票所有交易记录列表，每个元素包含：
  - `action`
  - `price`
  - `quantity`
  - `fee`
  - `trade_date`
  - `created_at`
  - `amount`
- `summary`（可选字典）：
  - `total_buy_quantity`
  - `total_sell_quantity`
  - `net_quantity`
  - `total_buy_amount`
  - `total_sell_amount`
  - `current_position_quantity`：来自 `positions` 表的数量（若存在）
  - `current_cost_price`：来自 `positions` 表的成本价（若存在）

#### 5.4 页面结构建议

- 页头：
  - 标题：`{{ stock_code }} {{ stock_name }}`。
  - 一行简要汇总：当前持仓数量、成本价、历史买入/卖出总金额等。
- 主体表格：
  - 列：日期、方向（买入/卖出）、价格、数量、手续费、金额、创建时间。
- 页脚：
  - 返回按钮：回到 `/positions/transactions` 或 `/positions`。

### 六、与阶段 2b.2 验收条目对照

#### 6.1 “添加操作”按钮与表单

- **按钮**：
  - 在现有 `/positions` 页面（`positions.html`）中增加“添加操作”按钮，指向 `/positions/trade`。
- **表单能力**：
  - 支持录入：股票代码、买入/卖出、价格、数量（可选：日期、手续费、名称）。
  - 提交后：
    - 在事务中写入 `transactions`；
    - 自动更新 `positions` 的数量与成本价；
    - 成功则重定向回持仓列表或历史页面。

#### 6.2 历史池子与单票明细

- **历史池子页面**
  - URL：`/positions/transactions`
  - 功能：基于 `transactions` 表，展示所有股票的历史买卖记录，按时间倒序。
- **单票操作明细页面**
  - URL：`/positions/<stock_code>/history_page`
  - 功能：展示某只股票的所有历史操作记录，并可选提供聚合统计与当前持仓对照。

### 七、后续扩展方向（预留给阶段三）

在上述极简设计基础上，未来“阶段三：仓位管理系统”可以进一步扩展：

- 对 `portfolio` 表引入真实资金账户逻辑（初始资金、可用资金、总市值）。
- 买入/卖出时，对资金余额做约束与校验。
- 引入手续费率、印花税等规则，取代单次录入的 `fee`。
- 加入更完整的仓位管理逻辑（单票仓位上限、整体仓位控制、止损止盈等）。

这些扩展可以在不破坏当前接口结构的前提下，平滑演进。
